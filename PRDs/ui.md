# Windows 本地翻译软件 UI 设计文档

## 一、文档目的

本文件用于指导 Codex 进行 UI 设计与前端实现，覆盖：

* 信息架构
* 页面布局
* 交互流程
* 状态设计
* 组件清单
* 视觉规范建议
* 页面草图（Markdown 线框图）

产品定位：

* 平台：Windows 本地桌面端
* 框架：Tauri
* 目标：轻量、响应快、本地优先、适合个人使用
* 核心能力：文本 / 图片 / 文档翻译，支持本地 LM Studio 与 OpenAI 兼容接口

---

## 二、设计目标

### 2.1 设计原则

1. **轻量优先**：界面简洁，不做复杂后台系统风格。
2. **翻译效率优先**：用户应在 1~2 步内完成输入、翻译、复制。
3. **可扩展**：后续能自然扩展到图片、文档、沉浸式翻译。
4. **低学习成本**：用户首次打开即可理解页面结构。
5. **本地工具感**：强调稳定、直接、少打扰。

### 2.2 设计关键词

* 极简
* 专注
* 轻桌面
* 专业但不过度商务化
* 快速响应

## 四、主翻译页设计

## 4.1 页面目标

用户在单一主界面完成：

* 选择输入类型
* 输入待翻译内容
* 选择目标语言
* 发起翻译
* 查看/复制/导出结果

## 4.2 页面布局结构

建议使用三段式结构：

* 顶部工具栏
* 中部双栏主工作区
* 底部状态栏

### 4.2.1 顶部工具栏

包含：

* 应用 Logo / 名称
* 输入类型切换（文本 / 图片 / 文档）
* 源语言选择（自动检测 / 中文 / 英文 / 俄语）
* 目标语言选择（中文 / 英文 / 俄语）
* 模型选择器
* “翻译”主按钮
* 更多操作菜单

### 4.2.2 中部主工作区

左右双栏：

* 左侧输入区
* 右侧输出区

### 4.2.3 底部状态栏

显示：

* 当前模型连接状态
* 最近一次翻译耗时
* 分段数 / token 提示
* 流式输出状态

## 五、主翻译页详细模块说明

## 5.1 顶部工具栏

### 5.1.1 Logo区

* 左上角显示应用名：`Local Translate`
* 可附带小字：`Powered by OpenAI-compatible Models`

### 5.1.2 输入类型切换

使用 Tabs：

* 文本
* 图片
* 文档

交互要求：

* 切换类型后保留当前草稿
* 不同类型的输入区展示不同控件

### 5.1.3 语言选择器

建议两个 Select：

* 源语言：默认“自动检测”
* 目标语言：默认“中文”

交互要求：

* 不允许源语言与目标语言完全一致时直接翻译，需提示
* 支持一键交换语言（Swap）

### 5.1.4 模型选择器

展示：

* 当前 Provider（如 LM Studio）
* 当前模型名（如 qwen3.5 35B）

交互要求：

* 下拉切换模型
* 异常时显示红点/告警 icon
* 鼠标悬停可看 baseURL 与连接状态

### 5.1.5 翻译按钮

主按钮文案：

* 默认：`开始翻译`
* 翻译中：`翻译中...`

状态：

* 空输入禁用
* 模型未连接禁用
* 翻译中支持取消

---

## 5.2 左侧输入区

## 5.2.1 文本输入模式

控件：

* 大文本框
* 工具行（粘贴、清空、字数统计、示例填充）

交互：

* 支持 Ctrl+V 粘贴
* 支持拖拽文本
* 支持自动高度/固定高度滚动
* 支持快捷键触发翻译（Ctrl+Enter）

状态：

* 空状态：提示用户输入待翻译文本
* 输入状态：显示字符数、估算分段数
* 长文本状态：显示“将自动分段翻译”提示

## 5.2.2 图片输入模式

控件：

* 拖拽上传区域
* “选择图片”按钮
* “粘贴截图”按钮
* 图片预览区

交互：

* 支持 PNG / JPG / JPEG / WEBP
* 上传后显示缩略图
* 支持删除当前图片
* 支持重新选择图片

状态：

* 未上传：空态插画/文本提示
* 已上传：显示图片与识别说明
* 翻译中：显示“正在调用多模态模型识别并翻译”

## 5.2.3 文档输入模式

控件：

* 拖拽上传区域
* “选择文档”按钮
* 文件信息卡片

支持：

* TXT
* DOCX
* PDF（后续若解析能力不足，可先标注实验性）

交互：

* 上传后显示文件名、大小、页数/段落数（能提取则显示）
* 显示“将自动提取文档文本并翻译”

---

## 5.3 右侧输出区

## 5.3.1 输出头部工具栏

包括：

* 显示模式切换：仅译文 / 双语对照
* 复制结果
* 导出结果
* 保存到历史
* 重新翻译

## 5.3.2 输出内容区

### 模式A：仅译文

只显示翻译结果，适合快速复制。

### 模式B：双语对照

按段落显示：

* 原文在上
* 译文在下

建议每段使用 Card 分组，方便阅读。

### 模式C：流式输出

翻译过程中逐段或逐行追加：

* 已完成片段显示正常
* 当前片段显示 typing/loading 状态

## 5.3.3 输出状态提示

包括：

* 翻译完成
* 翻译失败
* 模型超时
* 结果为空

错误文案示例：

* 当前模型连接失败，请检查 LM Studio 服务是否可访问
* 翻译超时，请稍后重试或切换模型

## 八、交互流程设计

## 8.1 文本翻译流程

1. 用户打开主页面
2. 默认进入“文本”模式
3. 输入待翻译文本
4. 选择目标语言
5. 点击“开始翻译”
6. 输出区流式显示结果
7. 用户复制/保存/导出

## 8.2 图片翻译流程

1. 切换到“图片”模式
2. 上传或粘贴图片
3. 点击翻译
4. 系统将图片发给多模态模型
5. 输出区显示识别后的译文

## 8.3 文档翻译流程

1. 切换到“文档”模式
2. 上传文件
3. 系统解析文本并提示分段
4. 用户点击翻译
5. 输出区按段落显示结果

* 已输入未提交

## 9.3 输出区状态

* 错误信息

## 十四、给 Codex 的实现提示

1. 优先按照 `AppShell / Toolbar / InputPanel / OutputPanel / Settings` 拆组件。
2. 所有输入模式使用统一数据结构，避免后续维护成本上升。
3. 翻译结果展示组件要支持文本、结构化段落、流式刷新三种状态。
4. 设置页中的模型配置要单独封装为 `ProviderConfigForm`。
5. 主窗口布局建议使用 CSS Grid，避免后期扩展困难。
6. 所有按钮、输入框、Tabs、Dialog、Drawer 尽量复用统一组件库。

## 十五、推荐技术与组件方案（供 Codex 直接采用）

### 15.1 总体推荐

* 桌面框架：Tauri
* 前端框架：React + TypeScript + Vite
* 样式方案：Tailwind CSS
* 组件库：shadcn/ui
* 状态管理：Zustand
* 服务请求：TanStack Query
* 表单管理：React Hook Form + Zod
* 图标库：Lucide React
* 动效：Framer Motion（仅少量使用）
* 文件拖拽上传：react-dropzone
* 文本编辑增强：原生 textarea 起步，后续可升级

### 15.2 原因

* 都是成熟方案，社区大，文档多，适合 Codex 自动生成与持续维护。
* React + TypeScript 对组件化拆分最友好。
* shadcn/ui 不是重型黑盒，方便本地桌面项目做定制。
* Zustand 比 Redux 更轻，适合个人工具。
* TanStack Query 非常适合处理翻译请求状态、缓存与重试。
* React Hook Form + Zod 可以让设置页表单稳定、易校验。

### 15.3 不建议首版使用

* 不建议首版上 Redux
* 不建议首版上 Ant Design
* 不建议首版上 Electron
* 不建议首版做复杂富文本编辑器
