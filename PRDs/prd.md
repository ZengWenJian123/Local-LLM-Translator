# 一、产品概述

## 1.1 产品背景

用户希望在Windows本地运行一个轻量级翻译工具，支持接入本地或远程大语言模型（兼容OpenAI API格式，如OpenAI、阿里云百炼、DeepSeek等），用于文本、图片、文档的多语言翻译。

## 1.2 产品目标

* 提供统一的翻译入口（文本 / 图片 / 文档）
* 支持多模型切换（本地/远程）
* 提供高质量、可控的翻译体验
* 低门槛部署，适用于个人或轻办公场景

## 1.3 目标用户

* 开发者（需要快速接入多模型）
* 翻译需求频繁的用户
* 学习外语用户
* 内容创作者

# 二、核心功能设计

## 2.1 输入模块

### 2.1.1 文本输入

* 多行文本输入框
* 支持粘贴（Ctrl+V）
* 支持拖拽文本
* 实时字数统计

### 2.1.2 图片输入（OCR）

* 支持拖拽图片
* 截图识别（快捷键触发）
* 内置OCR（可选本地OCR或云OCR）
* OCR结果可编辑

### 2.1.3 文档输入

支持格式：

* PDF
* Word（.docx）
* TXT

功能：

* 自动提取文本
* 分段处理
* 保持结构（标题/段落）

---

## 2.2 输出模块

### 2.2.1 翻译结果显示

* 分段对照（原文 / 译文）
* 支持切换显示模式：

  * 仅译文
  * 双语对照

### 2.2.2 目标语言

可选：

* 中文
* 英文
* 俄语

（后续可扩展）

### 2.2.3 结果操作

* 一键复制
* 导出（TXT / DOCX / Markdown）
* 保存历史记录

---

## 2.3 模型接入模块（核心能力）

### 2.3.1 支持模型类型

* 本地模型（如通过Ollama / 本地推理服务）
* 远程API（OpenAI兼容格式）

### 2.3.2 配置项

* API Base URL
* API Key
* Model Name
* Temperature
* Max Tokens

### 2.3.3 模型管理

* 多模型列表
* 一键切换
* 测试连接
* 默认模型设置

---

## 2.4 翻译引擎设计

### 2.4.1 Prompt模板

支持用户自定义：

* 翻译风格（正式/口语）
* 是否保留术语
* 是否解释

示例：
"请将以下内容翻译为{目标语言}，保持专业术语一致，并输出简洁结果："

### 2.4.2 分段翻译策略

* 长文本自动切分
* 并发翻译
* 合并结果

### 2.4.3 上下文记忆

* 保持同一任务术语一致
* 支持“会话翻译模式”

# 九、技术实现约束（基于确认信息）

## 9.1 技术选型

* 桌面框架：Tauri（轻量、性能优）
* 前端：React / Vue（建议React）
* 后端：Rust（Tauri内置）或Node sidecar

## 9.2 本地模型接入

* 使用已有LM Studio服务
* Base URL：[http://192.168.20.10:1234](http://192.168.20.10:1234)
* 协议：OpenAI兼容

### 接口示例

```
POST /v1/chat/completions
```

### 默认模型

* qwen3.5 35B

## 9.3 OCR方案

* 不使用传统OCR（如PaddleOCR）
* 统一使用多模态大模型进行OCR + 翻译

流程：
图片 → Base64 → LLM → 文本提取 + 翻译

## 9.4 沉浸式翻译（本地实现）

### 实现方式

* WebView注入JS
* 替换DOM文本
* 覆盖原文

### 功能点

* 页面全文翻译
* 段落级翻译
* 鼠标悬停翻译

# 十三、最终总结

该产品定位为：
👉 本地优先 + 多模型接入 + 轻量翻译工具

核心优势：

* 可控（本地模型）
* 可扩展（OpenAI协议）
* 高效率（快捷翻译 + 沉浸式）

建议开发路径：
👉 先做“文本翻译工具 + 模型接入”
👉 再做“生产力工具（快捷翻译 + 悬浮窗）"
